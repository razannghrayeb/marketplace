1. Executive Diagnosis ΓÇö WhatΓÇÖs Broken Today
After reading every file in the current codebase, here is an honest senior-engineering assessment of the gaps between what you have and what a real AI outfit assistant should do.
1.1 Category Detection is Shallow
detectCategory() in completestyle.ts works by keyword matching against CATEGORY_KEYWORDS. This breaks immediately on:
Products with no description (ΓÇ£Blue MaxiΓÇ¥ with no body text)
Multi-category products (ΓÇ£Blazer DressΓÇ¥, ΓÇ£Jogger ShortsΓÇ¥)
Non-English titles not covered by the Arabic/French keywords hardcoded
New category names coined by brands (ΓÇ£Utility Shirt-JacketΓÇ¥, ΓÇ£Ribbed Midi Co-ordΓÇ¥)
The ONNX extractAttributesSync fallback helps, but it is called after the keyword pass rather than as the primary signal. The category detection confidence is not surfaced to the caller in a way that changes retrieval strategy.
1.2 CATEGORY_PAIRINGS is a Static Lookup Table
The pairing rules in completestyle.ts (lines 386ΓÇô720) are hand-written. This means:
A tshirt always pairs with [&quot;jeans&quot;, &quot;pants&quot;, &quot;shorts&quot;, &quot;leggings&quot;] ΓÇö regardless of whether the shirt is a fitted white minimal tee (ΓåÆ tailored trousers, loafers) or an oversized band tee (ΓåÆ joggers, chunky sneakers).
Style nuance is lost at the pairing step, before retrieval even begins.
No feedback loop: a pairing that consistently produces zero clicks is never removed.
learnedPairings.ts and learnedCompatibility.ts exist but are not integrated into the main completeMyStyle() flow ΓÇö they write to a category_compatibility table that is never read by the outfit engine.
1.3 OpenSearch Retrieval is Keyword-Only
findProductsForCategory() builds an OpenSearch bool query using title keywords. This means:
A ΓÇ£navy slim-fit chinoΓÇ¥ will not match a query for ΓÇ£pantsΓÇ¥ if the product title says ΓÇ£trousersΓÇ¥
Color boosts are added as should clauses on the title field ΓÇö fragile and easily gamed
No semantic/embedding-based retrieval at the candidate-generation step
No reranking of the initial 2├ùN candidate pool before scoring
1.4 Scoring is a Hand-Tuned Point System
The scoring pipeline (lines 1200ΓÇô1430) accumulates points from formality delta, occasion match, color harmony, CLIP similarity, etc. The weights (FORMALITY_WEIGHT, COLOR_WEIGHT, etc.) are hardcoded constants. Problems:
These weights were not learned from data; theyΓÇÖre guesses
The neural ranker predictRankerScoresWithFallback is called at the end and overrides everything ΓÇö but itΓÇÖs presented as a fallback, so any failure silently reverts to the broken point system
CLIP is called per-product with no batching, making the endpoint very slow for 20+ candidates
1.5 Wardrobe Integration is a Post-Processing Hack
mergeWardrobeOwnedIntoCompletion() in outfit.service.ts loads up to 50 wardrobe items and staples them onto the front of each bucket. It does not:
Use wardrobe items to influence retrieval (e.g., skip recommending pants if the user already has 6 pairs of jeans)
Use the userΓÇÖs wardrobe style to calibrate the style profile of the seed item
Surface ΓÇ£you already own the perfect XΓÇ¥ as a first-class recommendation type
Handle the case where the userΓÇÖs wardrobe item is a better match than anything in the catalog
1.6 No LLM in the Critical Path
The outfitSuggestion field is generated by generateOutfitSuggestion() ΓÇö a hardcoded string builder. It produces output like:
ΓÇ£For your navy tshirt, great for everyday wear. We recommend pairing with jeans / pants, sneakers / boots.ΓÇ¥
This is not an AI outfit assistant. It is an if/else sentence builder.
1.7 Cultural Adaptation Exists but is Disconnected
lifestyleAdapter.ts has rich cultural logic. learnedPairings.ts has a culturalContext field. But neither feeds into the main recommendation flow. Every user gets Western fashion defaults.

4. Layer 1: Smarter Category &amp; Attribute Detection
4.1 Problem
Keyword matching is brittle. ONNX extraction is called as a fallback but its confidence score is not used to route decisions.
4.2 New Detection Strategy: Confidence-Gated Cascade
// NEW: detectCategoryV2 - confidence-gated cascadeexport async function detectCategoryV2(  product: Product): Promise&lt;DetectionResult&gt; {  // Stage 1: ONNX extraction (fast, ~10ms)  const onnxResult = await extractAttributes(product.title + &#39; &#39; + (product.description || &#39;&#39;), {    useML: true,  });  // If ONNX is highly confident, trust it  if (onnxResult.confidence.category &gt;= 0.85) {    return {      category: mapOnnxToProductCategory(onnxResult),      confidence: onnxResult.confidence.category,      source: &#39;onnx&#39;,      attributes: onnxResult,    };  }  // Stage 2: Keyword scoring (medium confidence baseline)  const keywordResult = scoreByKeywords(product.title, product.description);  if (keywordResult.confidence &gt;= 0.75) {    return {      category: keywordResult.category,      confidence: keywordResult.confidence,      source: &#39;keywords&#39;,      attributes: onnxResult, // still use ONNX attributes even if category came from keywords    };  }  // Stage 3: LLM classification (slow, ~300ms, only when unsure)  // Only called for ambiguous products ΓÇö estimated 5ΓÇô10% of requests  const llmResult = await classifyWithLLM(product);  return {    category: llmResult.category,    confidence: llmResult.confidence,    source: &#39;llm&#39;,    attributes: { ...onnxResult, ...llmResult.attributes },  };}

4.4 Key Change: Propagate Confidence Downstream
The DetectionResult.confidence must flow through to buildStyleProfile, getPairingRules, and findProductsForCategory. Low-confidence detection should widen the retrieval query (more candidate categories) rather than fail silently.

5. Layer 2: Compatibility Engine Redesign
5.1 Problem
Static CATEGORY_PAIRINGS ignores style nuance. learnedCompatibility.ts exists but never integrates.
5.2 Style-Aware Pairing Resolution
The pairing rules should NOT just be category ΓåÆ [target categories]. They must be: category + style profile ΓåÆ [target categories, ranked by compatibility]
// NEW: getStyleAwarePairingsexport async function getStyleAwarePairings(  detectedCategory: ProductCategory,  styleProfile: StyleProfile,  options: {    userId?: number;    culturalContext?: string;    limit?: number;  }): Promise&lt;PairingRule[]&gt; {  // 1. Get base static rules  const staticRules = CATEGORY_PAIRINGS[detectedCategory] || FALLBACK_PAIRINGS;  // 2. Get learned rules (NOW ACTUALLY INTEGRATED)  const learnedScores = await getCategoryPairings(detectedCategory, {    culturalContext: options.culturalContext,    config: { minSampleCount: 5, minConfidence: 0.5, fallbackToStatic: true },  });  // 3. Build a merged, de-duped rule set  const mergedRules = mergePairingRules(staticRules, learnedScores);  // 4. Re-rank by style compatibility  // e.g., a formal shirt should rank &quot;heels&quot; &gt; &quot;sneakers&quot; as footwear pairing  const styleRanked = rankByStyleCompatibility(mergedRules, styleProfile);  // 5. Apply user wardrobe gaps (don&#39;t recommend what they don&#39;t need)  if (options.userId) {    return applyWardrobeGapFiltering(styleRanked, options.userId);  }  return styleRanked.slice(0, options.limit || 8);}function rankByStyleCompatibility(rules: PairingRule[], style: StyleProfile): PairingRule[] {  return rules.map(rule =&gt; {    // Boost rules whose categories are compatible with this style    const stylePenalty = rule.categories.some(cat =&gt; {      const catStyle = CATEGORY_STYLE_MAP[cat as ProductCategory];      if (!catStyle) return false;      // Penalize if formality delta is large (e.g., don&#39;t recommend heels with a hoodie)      const formalityDelta = Math.abs((catStyle.formality ?? 5) - style.formality);      return formalityDelta &gt; 4;    }) ? 0.5 : 1.0;    return { ...rule, styleBoost: stylePenalty };  })  .sort((a, b) =&gt; {    // Primary: original priority (1 &gt; 2 &gt; 3)    if (a.priority !== b.priority) return a.priority - b.priority;    // Secondary: style boost    return (b.styleBoost ?? 1) - (a.styleBoost ?? 1);  });}

async function applyWardrobeGapFiltering(  rules: PairingRule[],  userId: number): Promise&lt;PairingRule[]&gt; {  // Load wardrobe category counts  const wardrobeCounts = await getWardrobeCategoryCounts(userId);  return rules.map(rule =&gt; {    const totalOwned = rule.categories.reduce((sum, cat) =&gt; {      return sum + (wardrobeCounts[cat] ?? 0);    }, 0);    // User owns 3+ items in this category ΓåÆ downgrade priority    if (totalOwned &gt;= 3 &amp;&amp; rule.priority === 1) {      return { ...rule, priority: 2 as const, reason: rule.reason + &#39; (you own some already)&#39; };    }    // User owns 0 ΓåÆ slight boost    if (totalOwned === 0 &amp;&amp; rule.priority === 2) {      return { ...rule, priority: 1 as const, reason: rule.reason + &#39; (wardrobe gap)&#39; };    }    return rule;  });}

6. Layer 3: Retrieval Pipeline Overhaul
6.1 Problem
OpenSearch retrieval is title-keyword-only. No vector search at candidate generation. Products that match semantically but not lexically are never surfaced.
6.2 Hybrid Retrieval: Lexical + Dense Vector
For each pairing bucket, run TWO retrieval strategies and merge:
async function findProductsForCategoryV2(  targetCategories: string[],  seedProduct: Product,  seedEmbedding: number[],         // Pre-computed, reused across buckets  styleProfile: StyleProfile,  options: FindOptions): Promise&lt;Product[]&gt; {  // Strategy A: Lexical (current behavior, fast, high precision)  const lexicalResults = await findByKeywords(targetCategories, styleProfile, options);  // Strategy B: Dense vector search (semantic, finds &quot;trousers&quot; when query is &quot;pants&quot;)  const vectorResults = await findByEmbedding(seedEmbedding, targetCategories, {    limit: options.maxResults,    filters: {      categories: targetCategories,      available: true,      priceRange: options.priceRange,    },  });  // Merge with Reciprocal Rank Fusion  return reciprocalRankFusion([lexicalResults, vectorResults], {    k: 60,    limit: options.maxResults * 2,  });}function reciprocalRankFusion(resultSets: Product[][], options: { k: number; limit: number }): Product[] {  const scores = new Map&lt;number, number&gt;(); // productId ΓåÆ RRF score  const productMap = new Map&lt;number, Product&gt;();  for (const results of resultSets) {    results.forEach((product, rank) =&gt; {      const rrfScore = 1 / (options.k + rank + 1);      scores.set(product.id, (scores.get(product.id) ?? 0) + rrfScore);      productMap.set(product.id, product);    });  }  return Array.from(scores.entries())    .sort(([, a], [, b]) =&gt; b - a)    .slice(0, options.limit)    .map(([id]) =&gt; productMap.get(id)!);}
